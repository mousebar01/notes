# AGENTS.md

本仓库用于记录个人笔记、经验、心得、过程复盘与零散想法。

这不是正式产品仓库，重点不是“工程完整性”，而是：

- 低阻力记录
- 可持续整理
- 尽量保留原始思路
- 在需要时帮助作者把碎片变成结构化知识

## 仓库定位

Agent 在这里的主要任务：

- 帮助整理和润色已有笔记
- 将零散记录归档成更清晰的主题文档
- 提炼经验、总结共性、补充目录或索引
- 在不破坏原意的前提下提升可读性
- 帮助建立长期可维护的笔记结构

不应把这里当作代码优先的项目仓库处理。除非用户明确要求，否则不要为了“规范”而过度改写内容。

## 工作原则

1. 先保留原意，再优化表达。
2. 先让内容可被找回，再追求形式统一。
3. 碎片记录也有价值，不要轻易删除。
4. 如无明确要求，优先追加、整理、归类，而不是重写。
5. 对主观感受、个人判断、阶段性结论，保持作者语气，不要擅自改成过度客观的公文风。

## 写作与整理建议

新增或整理内容时，优先采用以下思路：

- 标题清楚，能反映主题
- 开头先说明这条笔记在记录什么
- 经验类内容尽量包含“背景 / 过程 / 结论 / 建议”
- 心得类内容可以保留个人表达，但尽量避免只有一句抽象结论
- 若内容来自某次具体实践，尽量补上时间、场景、问题和结果

适合补充的结构示例：

```md
# 标题

## 背景

## 发生了什么

## 我的判断

## 结论 / 经验

## 后续可改进
```

不要求每篇都严格套模板，但整理时可以往这个方向靠。

## 内容处理规则

- 不删除作者原始记录，除非用户明确要求
- 对明显更好的版本，优先在原文基础上整理，而不是覆盖掉原始表达
- 若内容较乱，可新增“整理版”或“总结版”小节
- 若多个笔记主题相近，可在保留原文的前提下合并并建立索引
- 若信息不完整，可标注“待补充”，不要擅自编造

## 文件组织建议

当前仓库可以逐步演化为以下结构，但不必一次性建全：

```text
index/        导航页、目录、专题索引
topics/       通用知识笔记
projects/     长期项目、产品和系统设计
reference/    外部资料原文
inbox/        临时记录、速记、碎片想法
reviews/      复盘、总结、阶段回顾
archive/      已整理归档但仍需保留的旧内容
```

如果仓库暂时还是平铺结构，Agent 也应先尊重现状，仅在用户同意或确有必要时再批量整理。

## 命名建议

- 文件名优先清晰直白，避免过度追求花哨
- 可以使用中文文件名
- 若是阶段性记录，可加入日期，例如 `2026-06-03-关于笔记系统的想法.md`
- 若是长期主题，可使用稳定名称，例如 `LLM 学习心得.md`
- 在归档、合并或建立专题编排时，必要情况下可以重命名文件，使标题、文件名和目录关系更一致；但应保留原始语义，并同步更新相关索引、目录和链接。

## Agent 行为边界

- 不擅自删除内容
- 不擅自大规模移动文件
- 不把个人笔记改造成企业文档风格
- 不为了统一格式而牺牲信息密度
- 不将不确定内容写成确定事实

当需要做较大整理时，优先采用以下方式：

- 新建索引文件
- 新建整理稿
- 提出可执行的归档建议

而不是直接重构整个仓库。

## 适合 Agent 主动做的事

- 补标题
- 分段
- 提炼小结
- 增加目录
- 合并重复内容
- 给零散笔记加上上下文说明
- 把实践记录提炼为经验条目
- 帮用户把一段口语化描述整理成可复用笔记

## 编辑风格

- 使用 Markdown
- 语言以自然、清楚、耐读为主
- 可以保留一定个人口吻
- 少用夸张措辞
- 重点是“真实、有用、以后看得懂”

## 一个简单判断标准

如果一次编辑完成后，能更容易回答下面这些问题，就说明整理是有效的：

- 这条笔记在说什么？
- 为什么当时要记它？
- 它和过去的哪些经验有关？
- 未来我还能怎么用上它？

## 默认协作方式

当用户请求帮助时，Agent 默认按以下顺序工作：

1. 先理解这条笔记要保留的核心意思
2. 再判断是应该润色、补结构、合并，还是仅轻微整理
3. 尽量用最小改动提升可读性
4. 若涉及较大结构调整，先给出建议方案

这个仓库首先服务于“长期积累”，其次才是“整齐”。
