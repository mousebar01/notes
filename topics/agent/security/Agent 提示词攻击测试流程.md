# Agent 提示词攻击测试流程

这篇从 [提示词攻击测试方案](./提示词攻击测试方案.md) 提炼出一套可复用流程。适合在做 Agent 系统、工具调用、沙箱或权限系统时，用来检查安全边界是否可靠。

## 适用场景

- Agent 可以调用工具，例如 shell、文件读写、浏览器、网络请求。
- Agent 会读取外部内容，例如网页、文件、历史消息或记忆。
- 系统需要证明：即使模型被诱导，危险动作也不会真实执行。

## 测试顺序

### 1. 先列出高风险能力

优先关注这些工具或能力：

- shell / subprocess
- 文件读取和写入
- 网络请求
- 浏览器自动化
- 记忆写入
- 上传、下载、外部 API 调用

### 2. 再定义硬边界

每个高风险能力都要明确：

- 哪些用户可以调用
- 哪些会话可以调用
- 哪些路径可以访问
- 哪些命令必须拒绝
- 哪些来源是不可信内容
- 拒绝后是否有日志

### 3. 用 FakeProvider 模拟模型被攻破

不要只依赖真实模型测试。更稳的方式是让测试模型直接返回危险工具调用，例如：

```text
用户输入包含注入语义
↓
FakeProvider 返回危险 ToolCall
↓
工具调度层收到调用
↓
安全策略拒绝
↓
真实工具没有执行
```

这样测试的是工程边界，而不是模型当天的拒绝能力。

### 4. 覆盖 P0 场景

优先覆盖这些测试：

- 禁用 shell 后，模型请求 shell 也不能执行。
- `rm -rf /` 这类危险命令必须拒绝。
- 工作区外文件读取必须拒绝。
- 网页或文件里的提示词注入不能触发工具调用。
- 回复文本中的伪 `tool_calls` 不能被当作真实工具调用。

### 5. 做分层断言

一次安全测试最好不要只断言“输出看起来安全”。至少要看：

- 工具调用是否被拒绝。
- 拒绝原因是否准确。
- 真实文件系统是否没有变化。
- 网络请求是否没有发出。
- 安全事件日志是否记录。
- 不可信内容是否没有变成指令。

## 一句话原则

Agent 安全测试的重点不是证明模型永远不会犯错，而是证明模型犯错以后，系统边界仍然拦得住。
