# SSH 端口转发与代理配置原理解析

此笔记用于记录 SSH 端口转发（隧道）的本质、为什么它不会自动接管网络流量，以及如何配置应用程序使其正确使用代理通道。

## 1. 核心结论一句话总结

> **SSH 端口转发只提供网络路径，而代理配置（如环境变量、软件设置）负责指引程序走这条路径。**

---

## 2. 流量流向对比

当你使用 SSH `RemoteForward` 把本地 Clash 的 `7897` 端口转发到服务器的 `17897` 端口时：

```text
服务器 127.0.0.1:17897 (SSH 隧道入口)  <== SSH 隧道 ==>  本地 Clash 127.0.0.1:7897
```

### ❌ 默认情况：不配代理
即使转发成功，普通程序默认仍直接连接公网，完全不走隧道：
```text
普通程序 (例如 curl https://github.com) ──直接连接──> 目标网站 (被墙或超时)
```

###  使用代理：配置连接
只有当程序明确被告知连接服务器本地的 `17897` 端口时，流量才会流入隧道：
```text
程序 ──连接 127.0.0.1:17897 ──> SSH 隧道 ──> 本地 Clash ──> 节点 ──> 目标网站
```

---

## 3. SSH 转发与 Clash 本地代理的区别

### 为什么本地 Clash Verge 开启后很多软件直接可用？
因为 Clash Verge 提供了两种系统级接管模式：
1. **系统代理模式**：Clash 自动修改了操作系统的全局 HTTP/HTTPS 代理配置，浏览器等会自动读取。
2. **TUN 模式**：Clash 创建虚拟网卡，从网卡和路由层面直接劫持并重定向所有流量，实现透明代理。

### 为什么服务器上的 SSH 转发不会自动生效？
SSH `RemoteForward` 仅在服务器的 localhost 上**监听**一个端口，它**不会**：
- 修改服务器操作系统的代理设置。
- 修改系统的路由表。
- 创建虚拟网卡或截获现有连接。

---

## 4. 如何配置程序走代理通道

要想让程序走隧道，必须显式声明代理。

### 4.1 临时测试命令
```bash
curl -x http://127.0.0.1:17897 https://github.com
```

### 4.2 终端环境变量 (针对当前 Shell 及其子进程)
```bash
export HTTP_PROXY=http://127.0.0.1:17897
export HTTPS_PROXY=http://127.0.0.1:17897
export ALL_PROXY=socks5h://127.0.0.1:17897

# 同时设置小写版本 (部分程序只认小写)
export http_proxy="$HTTP_PROXY"
export https_proxy="$HTTPS_PROXY"
export all_proxy="$ALL_PROXY"
```
> [!TIP]
> 可以将上述配置写入服务器的 `~/.bashrc` 或 `~/.zshrc` 中，并执行 `source ~/.bashrc` 使其永久对该用户的终端生效。

### 4.3 VS Code 远程开发设置
在 `settings.json` 中配置：
```json
"http.proxy": "http://127.0.0.1:17897"
```

---

## 5. 形象比喻

- **SSH 端口转发**：相当于在山里修通了一条秘道，但没有更新导航地图。
- **代理配置（如环境变量）**：相当于修改了导航，指引车辆（网络请求）驶入这条秘道。
- **不配置代理**：车辆依然会沿着默认大路行驶，最终在关口被堵截。
