# 工程程序设计经验总结

这一路讨论下来，我有一个很重要的认知转变：

最开始看 Agent 代码时，我关注的是：

- `ConversationEngine` 好在哪里？
- `TurnRunner` 好在哪里？
- 为什么这样拆？

但后来慢慢发现，光看代码本身，其实很难判断好坏。

于是我们开始继续往下挖。

## 1. 从“功能”到“职责”

一开始看到：

```python
class ConversationEngine:
    ...
```

第一反应会觉得：

```text
这不就是转发给 TurnRunner 吗？
```

看不出精妙。

但往下拆后，才逐渐意识到每一层都在承担不同职责：

- `ConversationEngine` = 会话入口
- `TurnRunner` = 单轮编排
- `ContextStore` = 上下文准备
- `ReactiveReasoner` = 模型工具循环
- `Phase` = 生命周期节点
- `Hook` = 扩展机制

这时候才开始真正理解一件事：

> 好架构首先不是“技巧高级”，而是职责边界清晰。

## 2. 从“代码”到“变化”

后来我开始问：怎么判断代码写得好不好？

最直观的判断方式通常是看这些：

- 命名
- 风格
- 目录结构

但这些只能提供表面印象，不能真正说明质量。

真正的问题其实是：

```text
明天需求变了怎么办？
```

比如：

- 换模型
- 换 Memory
- 新增 Hook
- 新增 Tool
- 支持 MCP

这时候才意识到：

> 代码质量的本质，不是“现在能不能跑”，而是“变化时成本有多高”。

## 3. 从“变化”到“耦合”

再往下看，就会碰到一个更根本的问题：

- 为什么有的代码容易改？
- 为什么有的代码一改全崩？

根源通常就在耦合。

例如：

高耦合写法：

```python
self.reasoner = ReactiveReasoner()
```

低耦合写法：

```python
self.reasoner = deps.reasoner
```

这里的区别不只是“写法不同”，而是设计上的可替换性完全不同。

于是就能串起这样一条链：

```text
低耦合
↓
易构造
↓
易测试
↓
易维护
```

很多工程上的优点，其实不是彼此独立的，而是一环扣一环。

## 4. 从“测试”到“设计”

这是很关键的一步。

一开始我会觉得：测试是不是最难的部分？

后来慢慢发现，测试不是一个孤立技能。

测试本质上是在反向审视设计。

看到一个类时，不要先问：

```text
它是干什么的？
```

而是先问：

```text
我怎么测试它？
```

一旦这么问，很多设计问题会立刻暴露出来：

- 输入是什么？
- 输出是什么？
- 依赖是什么？
- 边界在哪里？

这时候才真正开始理解架构，而不只是“看懂代码流程”。

## 5. 重新理解“边界”

这是这次讨论里我最大的收获之一。

以前我对“边界”这个词的理解比较模糊，后来慢慢梳理出来：边界不是一个单一概念，至少有三层。

### 5.1 正常边界

最基础的一层是输入和输出。

```text
输入
↓
输出
```

例如：

```python
ContextStore.prepare(...)
```

输入消息，输出 `ContextBundle`。

### 5.2 异常边界

系统不只要处理正常情况，还要处理失败路径，例如：

- 非法输入
- Tool 失败
- LLM 超时
- Memory 崩溃

### 5.3 变化边界

还有一层常常更重要：未来替换和扩展时，系统能不能扛住。

例如：

- 换 Provider
- 换 Memory
- 换 ToolRegistry

如果一换就全身联动，说明边界其实没有建立起来。

所以我最后得到一个更完整的定义：

> 边界 = 模块对外的承诺 + 可能失败的地方 + 未来可能变化的地方。

## 6. 重新理解测试

后来再回头看测试代码，理解又变了。

一开始以为测试是在验证：

```text
功能是否正确
```

后来发现这只说对了一部分。

更准确地说，测试是在保护边界。

例如：

- `Foundation Types` 保护的是数据结构契约
- `ToolDiscoveryState` 保护的是状态管理规则
- `ReactiveReasoner` 保护的是模型工具循环
- `Lifecycle` 保护的是 Hook 扩展机制

所以可以把这个认知浓缩成一句话：

> 测试不只是测功能，测试是在保护设计。

## 7. 重新理解“工程”到底在做什么

最后一个很本质的问题是：

> 工程上是不是不只是写正常流程？

答案是肯定的。

甚至可以说：

```text
正常流程代码
可能只占 20%

剩下 80%
都在处理边界
```

例如：

- 用户输入错了怎么办？
- 模型超时怎么办？
- 模型乱调用工具怎么办？
- 工具失败怎么办？
- 工具无限循环怎么办？
- 上下文爆炸怎么办？
- 消息发送失败怎么办？
- 未来换模型怎么办？

这些问题往往才是工程工作的主体，而不是“happy path 写完就结束”。

## 最终总结

如果把这次讨论压缩成一句话，那就是：

> Demo 的目标是让功能跑起来。
>
> 工程的目标是让系统在变化、失败、扩展、协作的情况下仍然稳定运行。

所以以后再看 Agent 代码，不要先问：

```text
这个函数写得聪不聪明？
```

而要先问：

- 职责边界是否清晰？
- 依赖是否容易替换？
- 是否容易构造？
- 是否容易测试？
- 异常情况是否考虑？
- 未来变化是否容易支持？

当开始用这些问题审视代码时，就已经在从“会写代码的人”，逐渐转向“会设计系统的人”。

而这恰恰也是目前 AI 最难稳定替代、最有工程价值的能力之一。

## 一句话提醒自己

以后判断一段代码，不只看它能不能跑。

更要看它在变化、失败、扩展和协作到来时，能不能依然稳得住。
