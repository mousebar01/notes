看完了。除了 MemCell boundary，当前真正用到的算法大概有这几条：

**1. Ingest 标准化算法**

在 `src/mira/plugins/memory/ingest.py`。

作用是把当前 turn 变成 `CanonicalMessage`，并生成确定性 `message_id`：
`sha256 前 8 位` 是一个很短的内容指纹。

`sha256` 是一种哈希算法：给它一段文本，它会算出一串固定长度的十六进制字符串。比如：

```text
session_id + timestamp + role + text
  -> sha256
  -> a3f91c2e7b8d...
```

这里“前 8 位”就是只取开头 8 个字符：

```text
a3f91c2e
```

在我们这里它用来让 `message_id` 更稳定、更不容易撞：

```text
m_<session_id>_<timestamp_ms>_<index>_<sha256前8位>
```

假设同一个 session、同一个时间戳、同一个 index 下，用户消息内容不同，那么最后这个 hash 后缀也会不同。

例子：

```text
你好
-> m_sess_xxx_1781445475000_0_f8b29ca1

你还记得之前说了什么？
-> m_sess_xxx_1781445475000_0_7aa3e9d2
```

所以它不是加密，也不是为了安全，只是一个**内容指纹/去重标识**。  
取前 8 位是为了 ID 不要太长。这个长度有小概率碰撞，但在单个 session 的消息 ID 场景里通常够用；如果特别追求稳，可以取前 12 或 16 位。

```text
sha256(session_id + timestamp_ms + index + role + text)[:8]
```

然后 `merge_dedupe_sort()` 把：

```text
buffered tail + fresh messages
```

按 `message_id` 去重，再按 timestamp 排序。

**2. Markdown entry-level diff 算法**

在 `src/mira/plugins/memory/cascade.py:98` 起。

这个是同步 Markdown 到 DB / LanceDB 的核心：

```text
读整个 md 文档
  -> extract_entries(document)
  -> 对每个 entry 算 entry_content_hash
  -> 和 SQLite 里的 entry_state 对比
  -> 只对 changed entry 重新 derive_cells
  -> 未变化 entry 复用 old cells
  -> deleted entry 删除对应 cell
```

所以现在不是整篇都重新 embedding 了，已经是 entry 级 diff。  
但它仍然会解析整篇 Markdown，因为要知道哪些 entry 存在、哪些被删了。

**3. Cell 派生算法**

在 `src/mira/plugins/memory/cascade.py:246`。

每个 Markdown entry 会派生成一个或两个 `MemoryCell`：

```text
普通 cell:
  kind = episode / atomic_fact / foresight / user_profile / agent_case / agent_skill / idea_capture

如果 episode entry 带 memcell_id:
  额外生成 kind="memcell" 的 boundary cell
```

也就是说，一个 episode entry 通常会变成：

```text
memcell cell + episode cell
```

前者保留 boundary 结果，后者作为 episode 记忆进入检索。

**4. Embedding 算法**

在 `src/mira/plugins/memory/embedding.py`。

正式默认是 `OpenAICompatibleEmbedder`，也就是你配置的千问兼容接口：

```toml
[memory.embedding]
model = "text-embedding-v3"
base_url = "https://dashscope.aliyuncs.com/compatible-mode/v1"
```

`HashingTestEmbedder` 还在，但注释明确说是测试用，不是生产默认。

中文关键词 tokenize 也在这里：

```text
正则分词
中文 token 额外拆成单字和 bigram
```

这主要服务 BM25 / FTS 检索。

**5. LanceDB 检索算法**

在 `src/mira/plugins/memory/vector.py`。

有两路：

```text
sparse_search:
  LanceDB FTS 搜 text_tokens
  如果 FTS 失败，退到扫描 rows + token overlap 排分

vector_search:
  LanceDB 向量搜索
  distance 转 score = max(1.0 - distance, 0.0)
```

这里要注意：代码里 source 叫 `"BM25"`，但 fallback 的 `_rank_keyword_rows()` 是 token overlap，不是真正 BM25。LanceDB FTS 内部可能是 BM25/FTS 排序，但本地 fallback 不是。

**6. 混合检索融合算法**

在 `src/mira/plugins/memory/db.py:895` 和 `src/everalgo/rank/fusion.py`。

当前用的是 RRF：

```text
dense_candidates = vector_search(...)
sparse_candidates = sparse_search(...)
ranked_ids = rrf(dense_candidates, sparse_candidates, k=60)
```

RRF 公式是：

```text
score = Σ 1 / (60 + rank_i)
```

最后展示分数乘了 100：

```text
round(candidate.score * 100.0, 3)
```

**7. Cascade 聚类算法**

在 `src/mira/plugins/memory/cascade.py:181` 和 `src/everalgo/clustering/algorithm.py`。

每次 cells 变化后，`_refresh_clusters()` 会按：

```text
(user_id, kind)
```

把所有 memcell 重新聚类。

普通 kind 用几何聚类：

```text
cluster_by_geometry
cosine similarity >= 0.65
time_window_days = 7
```

合并方式是 weighted centroid：

```text
new_centroid = (old_centroid * old_count + new_vector) / total_count
```

`agent_case` 特殊一点：如果有 LLM provider，就用 `cluster_by_llm()`：

```text
先取 top-30 近邻
如果 top1 similarity >= 0.85，直接并入
否则让 LLM 判断并入哪个 cluster，或者返回 new
```

**8. OME 直接抽取算法**

在 `src/mira/plugins/memory/ome.py:23` 和 `:236` 起。

每个 episode entry 会入 5 个 OME job：

```text
extract_atomic_facts
extract_foresight
extract_agent_case
extract_user_profile
extract_agent_skill
```

直接抽取包括：

```text
AtomicFactExtractor
ForesightExtractor
AgentCaseExtractor
```

它们来自 `src/everalgo/user_memory` / `src/everalgo/agent_memory`，都是 LLM JSON 结构化抽取。

其中 `foresight` 有证据校验：

```python
evidence in source.text
```

也就是证据必须是原文连续子串。

**9. OME 聚类抽取算法**

在 `src/mira/plugins/memory/ome.py:341` 和 `:546` 起。

`user_profile` 从这些 kind 里找近邻：

```text
episode
atomic_fact
user_profile
```

`agent_skill` 从这些 kind 里找近邻：

```text
agent_case
agent_skill
```

具体做法是：

```text
source.text embedding
  -> 遍历已有 cluster members
  -> 读取每个 member 的 vector_json
  -> cosine(source_vector, member_vector)
  -> 去重
  -> score 降序
  -> 取 top 12
```

然后要求 `min_cluster_size=2`，才调用：

```text
ProfileExtractor
AgentSkillExtractor
```

**10. 当前没接入主路径的算法**

`src/everalgo/rank/` 里还有很多算法，比如：

```text
lr
vector_anchored
maxsim
agentic_retrieve
LLM rerank
case/profile/skill ranker
```

但 Mira 当前 `MemoryDatabase.search()` 主路径只用到了 `rrf()`，没有接入这些更复杂的 ranker。

一句话总结：现在主路径算法是：

```text
deterministic ingest
-> LLM boundary
-> Markdown entry diff
-> embedding
-> LanceDB sparse/vector
-> RRF fusion
-> geometry/LLM clustering
-> OME direct + cluster extraction
```

真正还需要警惕的是两个点：  
`BM25` 这个名字现在有点泛化，fallback 不是严格 BM25；另外 `_refresh_clusters()` 是 cells 变化后全量重聚类，不是增量维护 cluster 状态。
