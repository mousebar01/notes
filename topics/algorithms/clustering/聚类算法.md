对，当前是把**所有已经存在的 cell 重新拿出来聚类**，而且会分成很多类。重点是：**它不预先指定要分成几类**，而是边遍历边决定。

当前实现可以理解成：

```text
所有 memcell 表里的 cell
  ↓
按 (user_id, kind) 分组
  ↓
每组内部按 created_at 排序
  ↓
从早到晚一个个处理
  ↓
能归入已有 cluster 就归入
  ↓
不能归入就新建 cluster
```

举个例子，假设同一个用户有这些 `episode` cell：

```text
A: 我在设计记忆系统
B: Markdown 应该作为事实源
C: LanceDB 只做检索索引
D: 周末想去爬山
E: 明天要面试
F: 记忆系统里 OME 怎么做
```

算法不是说“我要分成 3 类”，而是这样跑：

```text
A 来了，没有已有 cluster
  -> 新建 cluster_1: [A]

B 来了，和 cluster_1 相似度高
  -> 并入 cluster_1: [A, B]

C 来了，也相似
  -> 并入 cluster_1: [A, B, C]

D 来了，和 cluster_1 不相似
  -> 新建 cluster_2: [D]

E 来了，和 cluster_1 / cluster_2 都不相似
  -> 新建 cluster_3: [E]

F 来了，和 cluster_1 相似
  -> 并入 cluster_1: [A, B, C, F]
```

最后自然形成：

```text
cluster_1: 记忆系统设计
cluster_2: 爬山计划
cluster_3: 面试
```

但代码里不会真的理解“记忆系统设计”这个名字，目前 label 只是：

```text
"{kind} cluster"
```

也就是说它有分组，但没有智能命名。

**当前具体实现**

在 `_refresh_clusters()` 里，它先从 SQLite 取出所有 cell：

```text
SELECT cell_id, user_id, kind, text, created_at, vector_json FROM memcell
```

然后按：

```text
(user_id, kind)
```

分组。

这个很重要，因为不同用户、不同记忆类型不会混在一起聚类。

比如：

```text
(default_user, episode)
(default_user, atomic_fact)
(default_user, user_profile)
(default_user, agent_case)
```

这些是分开聚类的。

所以不会出现：

```text
episode 和 agent_case 混成一类
```

也不会出现：

```text
用户 A 和用户 B 的记忆混成一类
```

**每个 cell 怎么判断归到哪个 cluster**

每个 cell 已经有一个 embedding 向量：

```text
vector_json = [0.01, -0.12, ...]
```

当前 cell 会被包装成一个临时 cluster：

```text
incoming cluster:
  id = cluster_<cell_id>
  centroid = 当前 cell 的向量
  count = 1
  last_ts = 当前 cell 时间
  preview = 当前 cell 文本前 200 字
  members = [cell_id]
```

然后拿这个 incoming cluster 去和已有 clusters 比较。

普通类型走：

```text
cluster_by_geometry()
```

它会在已有 clusters 里找最像的那个：

```text
cosine(new_vector, cluster_centroid)
```

并且要求两个条件：

```text
相似度 >= 0.65
时间距离 <= 7 天
```

如果找到了，就合并进去。

如果没有找到，就新建 cluster。

**合并时发生什么**

假设已有 cluster 有 3 个成员：

```text
cluster_1:
  count = 3
  centroid = 旧中心
  members = [A, B, C]
```

新 cell F 进来后，会更新中心：

```text
new_centroid = (old_centroid * old_count + new_vector) / (old_count + 1)
```

也就是：

```text
新中心 = 旧 cluster 的平均方向，加上新 cell 后重新平均
```

然后更新：

```text
count += 1
last_ts = max(old.last_ts, new.last_ts)
preview = 保留最后几个 preview
members = old.members + new.members
```

所以 cluster 不是固定的，它会随着成员加入而移动中心。

**为什么会形成多类**

因为只要某个 cell 和所有已有 cluster 都不够像，就会新建一类。

所以类的数量是自然增长出来的：

```text
相似的 cell 越多 -> 合并越多 -> cluster 少
主题越分散 -> 新建越多 -> cluster 多
```

当前没有类似：

```text
必须分成 5 类
必须每类至少 3 条
必须每类名字不同
```

这些限制。

**agent_case 特殊逻辑**

`agent_case` 不完全一样。

如果有 LLM provider，它会走：

```text
cluster_by_llm()
```

逻辑是：

```text
先用向量找 top-30 候选 cluster
如果 top1 similarity >= 0.85
  -> 直接并入，不问 LLM
否则
  -> 把新 case 和候选 cluster preview 给 LLM
  -> 让 LLM 返回并入哪个 cluster，或者返回 new
```

所以 `agent_case` 的聚类更偏“这个经验方法是否属于同一类任务/同一类做事经验”。

普通记忆只看向量相似度，`agent_case` 在模糊时会让模型判断。

**聚类结果存哪里**

聚完以后，每个 cluster 会写入 SQLite 的 cluster 表，大概包括：

```text
cluster_id
owner_id
kind
label
centroid_json
members
last_ts
preview
```

其中：

```text
cluster_id = cluster_<第一个 cell_id>
members = 这个 cluster 下面有哪些 cell
centroid_json = 这个 cluster 的中心向量
preview = 这个 cluster 里一些文本预览
```

注意：当前 cluster 是派生产物，不是 Markdown 事实源。  
Markdown 还是 source of truth；cluster 可以删掉重建。

**当前实现的特点**

优点：

```text
不用预先指定类别数量
新增主题会自然变成新 cluster
相似主题会自动合并
普通聚类不烧 token
可以从 Markdown / memcell 全量重建
```

问题：

```text
每次 cells 变化后是全量重聚类
cluster label 很粗糙，没有主题名
几何聚类依赖 embedding 质量和阈值
0.65 / 7天 这些参数仍然会影响效果
agent_case 的 LLM 聚类可能烧 token
```

一句话：

```text
当前聚类不是“指定分成几类”，而是“按时间顺序逐条 cell 归并；相似就进已有类，不相似就新建类”。
```

你感觉对了，这个聚类确实不只是“算个 cosine 相似度”那么简单。它实际是一个**全量重建的、按时间顺序在线归并的 cluster 构建过程**，有点像轻量版 BIRCH 的思路：每条 cell 进来时被当作一个 `count=1` 的小 cluster，再尝试并入已有 cluster。

完整过程是这样：

```text
Markdown entry 变化
  ↓
cascade 更新 memcell 表
  ↓
如果 cell 有变化
  ↓
_refresh_clusters()
  ↓
读取 memcell 表里所有 cell
  ↓
按 (user_id, kind) 分组
  ↓
每组内部按 created_at 排序
  ↓
逐条 cell 构造成 incoming cluster
  ↓
尝试并入已有 cluster
  ↓
不能并入则新建 cluster
  ↓
把最终 cluster / cluster_member 覆盖写回 SQLite
```

关键点有几个。

第一，它是**全量重聚类**，不是增量聚类。

代码里 `_refresh_clusters()` 会直接查：

```sql
SELECT cell_id, user_id, kind, text, created_at, vector_json FROM memcell
```

也就是每次有 cell 改动，它不是只处理那几条 changed cell，而是把当前所有 cell 重新跑一遍。

第二，它会先按：

```text
(user_id, kind)
```

分组。

所以这些不会混在一起：

```text
default_user + episode
default_user + atomic_fact
default_user + memcell
default_user + user_profile
```

这也是为什么你现在看到 `episode` 和 `memcell` 各有 5 个 cluster，它们其实是分开跑了两遍。

第三，每个 cell 会先被包装成一个临时 cluster：

```python
Cluster(
    id=f"cluster_{cell_id}",
    centroid=cell_vector,
    count=1,
    last_ts=created_at_ms,
    preview=[text[:200]],
    members=[cell_id],
)
```

这里的 `id=f"cluster_{cell_id}"` 很关键：  
如果它没有并入任何已有 cluster，它就会成为一个新 cluster，cluster id 就是第一条 cell 的 id 派生出来的。

第四，普通 kind 用 `cluster_by_geometry()`。

它不是和所有 cell 比，而是和“已有 cluster 的中心点”比：

```text
cosine(incoming.centroid, existing_cluster.centroid)
```

并且要求：

```text
相似度 >= 0.65
时间距离 <= 7 天
```

注意时间判断是：

```text
abs(incoming.last_ts - cluster.last_ts) <= 7 days
```

而 `cluster.last_ts` 会随着合并更新成最新成员时间。

第五，如果能并入，就不是简单追加，而是更新 cluster 中心：

```text
new_centroid =
  (old_centroid * old_count + incoming_centroid * incoming_count)
  / (old_count + incoming_count)
```

因为 incoming 默认 `count=1`，所以可以理解成：

```text
旧 cluster 的平均向量 + 新 cell 向量，再重新平均
```

同时还会更新：

```text
count
last_ts
preview
members
```

`preview` 只保留最后 5 个片段：

```python
preview=(existing.preview + new.preview)[-preview_cap:]
```

但 `members` 不截断，会一直追加。

第六，它是**顺序敏感**的。

因为它按 `created_at` 从早到晚处理，所以早期形成的 cluster 会影响后面 cell 的归属。  
这不是 KMeans 那种反复迭代到收敛的算法，也不会全局寻找最优划分。

也就是说：

```text
A 先建 cluster
B 并入 A
C 再拿自己和 AB 的中心比
```

而不是：

```text
A/B/C 全局一起优化
```

第七，`agent_case` 特殊。

普通 kind：

```text
cluster_by_geometry()
```

`agent_case` 如果有 LLM provider：

```text
cluster_by_llm()
```

它会：

```text
先按向量找 top-30 cluster
top1 similarity >= 0.85 -> 直接并入
否则 -> 发给 LLM 判断并入哪个 cluster，或新建
```

而且 `cluster_by_llm()` 这条路径里没有 7 天时间窗限制，主要靠 top-k 向量召回 + LLM 判断。

第八，落盘是覆盖式的。

聚完后，代码会对每个 cluster 调：

```python
upsert_cluster(...)
```

它会写 `cluster` 表，然后先删掉这个 cluster 旧的 members：

```sql
DELETE FROM cluster_member WHERE cluster_id=?
```

再重新插入当前 members。

最后还会：

```python
delete_clusters_not_in(active_cluster_ids)
```

把这次没有重新生成出来的旧 cluster 删除。

所以 cluster 表不是历史日志，而是当前快照。

第九，当前 `cluster_member.score` 不是相似度。

写入时是：

```python
member_scores = [(member_id, 1.0) for member_id in cluster.members]
```

所以现在每个 member 的 score 都是 `1.0`。  
这意味着后面 `get_cluster_members(... ORDER BY score DESC)` 其实没有真实排序意义。

第十，它还有一个设计上的副作用：`memcell` 和 `episode` 会重复聚类。

因为 episode entry 会派生：

```text
kind=memcell
kind=episode
```

于是同一批文本可能在两个 kind 下各自聚一遍。现在你看到的 `memcell` cluster 和 `episode` cluster 很像，就是这个原因。

所以一句话更准确地说：

```text
当前聚类 = 从 memcell 表全量读取所有 cell，按 user/kind 分组后，以 created_at 顺序做在线归并；普通类型用 cosine + 时间窗决定是否并入，agent_case 可用 LLM 判断；最终把 cluster 快照写入 SQLite。
```

它确实不算特别简单，主要复杂在“聚类算法本身 + SQLite 派生状态重建 + OME 后续读取 cluster”揉在一起了。
