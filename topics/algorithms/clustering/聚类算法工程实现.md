这篇是经典的 **CluStream** 论文，和刚才 EverMemBench 不一样：**这篇是真正提出算法框架的论文**。

论文题目是 **A Framework for Clustering Evolving Data Streams**，核心问题是：
数据流不断到来、不能反复扫描，而且数据分布会随时间变化。传统一次性聚类或者 one-pass k-means 会被“过时历史”支配，无法回答“最近一个月的簇是什么”“去年和现在的簇怎么变了”这类问题。

## 一句话概括

这篇论文提出 **CluStream**：

```text
在线阶段：维护很多细粒度 micro-cluster
离线阶段：按用户指定时间窗口，把 micro-cluster 聚成 macro-cluster
时间存储：用 pyramidal time frame 保存历史快照
```

它本质上是一个**流式数据的时间感知聚类框架**。

## 为什么不能直接用普通 k-means？

因为普通 k-means 或简单 one-pass clustering 有几个问题：

第一，数据流不能回头扫描。
第二，数据一直演化，早期数据可能已经不代表当前状态。
第三，如果两个簇早期被合并了，后面想再拆开就很难。
第四，用户可能想查不同时间窗口，比如最近 1 小时、最近 1 天、最近 1 年。普通算法无法高效支持。

所以论文把问题拆成两层：

```text
online micro-clustering：快速吸收数据，保存统计摘要
offline macro-clustering：用户查询时，再基于摘要做聚类
```

## 核心算法 1：Micro-cluster

Micro-cluster 是这篇论文最核心的表示。

它不是保存原始点，而是保存一个簇的统计量：

```text
CF2x：每个维度的平方和
CF1x：每个维度的和
CF2t：时间戳平方和
CF1t：时间戳和
n：点的数量
```

所以一个 d 维 micro-cluster 是：

```text
( CF2x, CF1x, CF2t, CF1t, n )
```

这个设计的关键是：它有**可加性**和**可减性**。
两个 micro-cluster 可以相加，表示合并；两个时间快照之间可以相减，得到某个时间窗口内的统计摘要。

这点非常重要。因为它让系统不需要保存所有原始数据，也能回答：

```text
最近 h 时间窗口内的数据形成了什么簇？
```

## 核心算法 2：Pyramidal Time Frame

如果每个时刻都保存一次 micro-cluster 快照，存储会爆炸。

所以论文提出 **pyramidal time frame**：
越近的历史保存得越密，越远的历史保存得越稀。

类似这样：

```text
最近几秒：高频快照
最近几分钟：稍稀疏
最近几小时：更稀疏
很久以前：只保留少量关键快照
```

论文证明，用这种方式可以用较少快照近似任意用户指定的时间窗口。例如它给了一个极端例子：如果数据流跑 100 年，按秒级粒度，在某些参数下也只需要很有限的快照数量；提高参数后可以把任意时间窗口近似到很高精度。

这个思想和长期记忆很像：**近事保细，远事保粗**。

## 核心算法 3：在线维护 micro-cluster

新数据点来了以后，算法大概这样处理：

```text
1. 找到离新点最近的 micro-cluster
2. 判断这个点是否落在该 micro-cluster 的边界内
3. 如果在边界内：吸收到这个 micro-cluster
4. 如果不在边界内：新建一个 micro-cluster
5. 如果 micro-cluster 数量超限：
   - 优先删除不活跃 / 过时的 micro-cluster
   - 如果不能删，就合并最近的两个 micro-cluster
6. 按 pyramidal time frame 定期保存快照
```

这里有个细节：
它不是简单删除“点数最少”的 micro-cluster，而是计算 **relevance stamp**，也就是这个 micro-cluster 最近是否还活跃。因为一个簇过去可能很大，但现在已经不活跃了，对当前聚类意义就不大。

这点对记忆系统也很有启发：
不是“用得少就删”，而是看它在当前时间窗口里是否还相关。

## 核心算法 4：离线 macro-cluster

用户查询时会给两个参数：

```text
h：时间窗口，例如最近 7 天
k：想要几个大簇
```

系统会找当前快照和过去某个快照，然后利用 micro-cluster 的**可减性**，得到这个时间窗口内的 micro-cluster 集合。

然后把这些 micro-cluster 当成 pseudo-points，再用修改版 k-means 聚成 k 个 macro-cluster。修改点包括：

```text
1. 初始化种子不是随机点，而是按 micro-cluster 的点数加权采样
2. 距离计算使用 micro-cluster 的中心
3. 更新中心时使用 weighted centroid
```

也就是说：

```text
原始数据点太多，不直接聚类；
先压缩成 micro-cluster；
查询时再把 micro-cluster 聚成 macro-cluster。
```

## 核心算法 5：演化分析

这篇论文还有一个重要功能：比较两个时间点的簇变化。

用户输入：

```text
t1：过去时间点
t2：当前或另一个时间点
h：比较窗口
```

系统比较：

```text
(t1 - h, t1) 的簇
和
(t2 - h, t2) 的簇
```

然后把变化分成三类：

```text
M_added：新增的 micro-cluster
M_deleted：消失的 micro-cluster
M_retained：保留但可能变化的 micro-cluster
```

这就能回答：

```text
有没有新簇出现？
有没有旧簇消失？
有没有簇的位置或性质发生变化？
```

论文专门说，micro-cluster 的 idlist 可以用来追踪这些变化。

## 实验结论

实验对比的是 CluStream 和 STREAM。论文使用了 KDD-CUP'99 网络入侵数据集、KDD-CUP'98 捐赠数据集，以及合成数据。结果显示，CluStream 在快速演化的数据流上明显优于 STREAM，尤其在网络入侵数据上，SSQ 指标可以低几个数量级。论文认为优势来自两个设计：pyramidal time frame 能近似任意时间窗口，micro-cluster 保存了足够细粒度的统计信息。

## 和你做长期记忆有什么关系？

这篇虽然是数据流聚类，不是 LLM memory，但它对你很有启发。

你现在关心的是长期记忆、主动关心、用户创意助手。CluStream 给了一个非常清晰的工程范式：

```text
不要把所有历史都平铺保存；
而是在线维护细粒度摘要；
按时间快照保存；
查询时按时间窗口重构；
再做高层聚合。
```

映射到记忆系统可以是：

```text
原始对话 / 浏览行为 = data stream
micro-cluster = 小型记忆单元 / 事件簇
pyramidal snapshots = 近细远粗的记忆快照
macro-cluster = 当前主题 / 用户阶段 / 创意方向
evolution analysis = 用户兴趣、项目方向、偏好变化
```

比如用户早期一直在聊“记忆系统”，后来转向“创意助手”，再后来收敛到“真实痛点、流程、竞品分析”。
如果用 CluStream 思路，你不是简单存一堆聊天记录，而是维护“主题簇随时间的演化”。

## 最值得借鉴的一句话

我觉得这篇论文对你最有价值的点是：

```text
长期系统不能只保存最终聚类结果，
而应该保存足够细粒度、可加可减、可按时间窗口重构的中间摘要。
```

这比单纯 embedding memory 更工程化。
如果你要设计 Mnemo 的记忆层，CluStream 的思想可以转成：

```text
MemoryStream：
在线阶段维护事件级 memory micro-clusters；
定期保存时间快照；
查询时按时间窗口重建相关记忆；
再聚合成当前用户状态、兴趣状态、项目状态。
```
