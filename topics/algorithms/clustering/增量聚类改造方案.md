# 增量聚类改造方案

这份方案用于把 CluStream 的 `online micro-cluster` 思想裁剪到 Mira 记忆系统中。

目标不是完整复现 CluStream，而是解决当前工程问题：

```text
当前：cell 变化 -> 全量读取所有 memcell -> 全量重聚类
目标：cell 变化 -> 只更新受影响 cluster
```

## 1. 设计结论

第一版采用：

```text
BIRCH-style Online Micro-Cluster
```

也就是：

```text
新 MemoryCell 进入
  ↓
找同 user_id + kind 下最近的 cluster
  ↓
用 cosine 判断是否能吸收
  ↓
能吸收则用 CF 统计量增量更新
  ↓
否则新建 cluster
```

这里参考 BIRCH 的核心不是完整 CF Tree，而是它的 `Clustering Feature` 统计量：

```text
N  = 成员数量
LS = 向量线性和，linear sum
SS = 向量平方范数和，sum of squared norms
```

删除或修改 cell 时，优先用 `N / LS / SS` 做增量扣减；只有数据异常时才局部重算受影响 cluster。

保留全量重建能力，用于维护和迁移：

```text
recluster_all()
  -> 清空 cluster / cluster_member
  -> 从 memcell 表全量重建 cluster
```

## 2. 和 CluStream 的对应关系

CluStream / BIRCH 里的概念可以映射成：

```text
data point          -> MemoryCell
micro-cluster       -> cluster 表中的一条记录
micro-cluster member-> cluster_member 表
macro-cluster       -> OME 派生出的 profile / skill / idea cell
snapshot            -> 后续可选的 cluster_snapshot
BIRCH CF            -> cluster 表里的 N / LS / SS / centroid
```

第一版只实现：

```text
online micro-cluster + BIRCH CF 统计量
```

暂不实现：

```text
pyramidal time frame
offline macro-kmeans
evolution analysis
```

原因是 Mira 已经有 Markdown source of truth，可以随时重建派生状态；第一版优先降低当前全量重聚类成本。

## 3. 当前问题

当前 `_refresh_clusters()` 的行为是：

```text
读取 memcell 表所有 cell
  ↓
按 (user_id, kind) 分组
  ↓
每组按 created_at 排序
  ↓
逐条在线归并
  ↓
覆盖写入 cluster / cluster_member
```

问题：

```text
1. 每次 cell 变化都全量重聚类。
2. agent_case 可能触发 LLM 聚类，和全量刷新组合后容易烧 token。
3. cluster_member.score 当前固定写 1.0，没有真实相似度。
4. memcell 和 episode 会重复聚类。
5. cluster 只是当前快照，没有 algorithm_version，后续迁移不够清晰。
```

## 4. 第一版改造范围

第一版做这些：

```text
1. 新增 IncrementalClusterer。
2. 让 replace_cells_for_entries() 返回 changed / deleted cell 信息。
3. cascade 中用增量聚类替代默认全量 _refresh_clusters()。
4. 保留 _refresh_clusters()，改名或定位为 rebuild_clusters()。
5. 默认不聚类 kind=memcell。
6. cluster_member.score 写入真实相似度。
7. cluster 表增加 algorithm_version 和 CF 统计量字段。
```

第一版暂不做：

```text
1. pyramidal snapshot。
2. offline macro-kmeans。
3. adaptive threshold。
4. pending / outlier buffer。
5. 默认 LLM refine。
```

## 5. 聚类对象范围

建议默认参与聚类的 kind：

```python
CLUSTERABLE_KINDS = {
    "episode",
    "atomic_fact",
    "foresight",
    "agent_case",
    "agent_skill",
    "idea_capture",
}
```

暂不默认聚类：

```text
memcell
user_profile
```

原因：

```text
memcell:
  这是 Mira 为了保留 boundary cell 额外引入的检索 kind，不是原始 memory doc type。
  当前它有两条来源：
    1. boundary detector 产出的 BoundaryMemCell 会直接写入 kind=memcell。
    2. episode Markdown entry 带 memcell_id 时，cascade 会再派生一个 kind=memcell。
  因此它更适合作为 boundary 追踪 / 检索对象，而不是 OME 聚类对象。
  如果同时聚类 memcell 和 episode，容易形成内容相近的镜像 cluster。

user_profile:
  本身是高层派生结果，继续聚类可能造成自我强化。
```

## 6. BIRCH CF 公式设计

BIRCH 里的 `CF` 可以直接用于我们的增量维护，但距离判断需要改成适合 embedding 的 cosine 版本。

### 6.1 CF 定义

对一个 cluster，设成员向量为：

```text
x1, x2, ..., xn
```

维护：

```text
N  = n
LS = x1 + x2 + ... + xn
SS = ||x1||^2 + ||x2||^2 + ... + ||xn||^2
```

cluster 的中心向量：

```text
centroid = LS / N
```

如果 embedding 是单位向量，那么：

```text
||xi||^2 ≈ 1
SS ≈ N
```

但仍然建议保存 `SS`，因为它后续可以用于计算半径、离散度和异常检测。

### 6.2 新增 cell

新 cell 的向量为 `x`。

如果它被吸收到已有 cluster：

```text
N'  = N + 1
LS' = LS + x
SS' = SS + ||x||^2
centroid' = LS' / N'
```

如果它不能被吸收，则新建 cluster：

```text
N  = 1
LS = x
SS = ||x||^2
centroid = x
```

### 6.3 删除 cell

删除旧 cell，旧向量为 `x`：

```text
N'  = N - 1
LS' = LS - x
SS' = SS - ||x||^2
centroid' = LS' / N'
```

如果：

```text
N' <= 0
```

则删除该 cluster。

### 6.4 修改 cell

修改 cell 等价于：

```text
删除旧 cell + 新增新 cell
```

如果旧向量为 `old_x`，新向量为 `new_x`，且确定它仍然留在同一个 cluster，可以写成：

```text
N'  = N
LS' = LS - old_x + new_x
SS' = SS - ||old_x||^2 + ||new_x||^2
centroid' = LS' / N'
```

但工程上建议把修改处理为两步：

```text
remove old cell from old cluster
assign new cell to best cluster
```

因为内容修改后，它不一定还应该属于原 cluster。

### 6.5 合并两个 cluster

如果后续需要合并两个 cluster：

```text
N  = N1 + N2
LS = LS1 + LS2
SS = SS1 + SS2
centroid = LS / N
```

这就是 BIRCH CF 的可加性。

### 6.6 cosine 吸收判断

BIRCH 原始论文更偏欧氏距离场景。Mira 的 MemoryCell 是 embedding，更适合用 cosine。

对新 cell 向量 `x` 和 cluster 中心 `centroid`：

```text
centroid_unit = normalize(centroid)
score = cosine(x, centroid_unit)
```

如果：

```text
score >= threshold(kind)
```

则吸收进该 cluster。

否则新建 cluster。

第一版不使用 BIRCH 原始半径阈值作为主判断，避免和 embedding cosine 语义冲突。

### 6.7 cohesion / dispersion

有了 `N / LS / SS`，可以计算 cluster 的方向一致性。

推荐记录或调试时计算：

```text
cohesion = ||LS|| / N
dispersion = 1 - cohesion
```

直觉：

```text
cohesion 越接近 1，成员方向越一致。
dispersion 越接近 0，cluster 越紧。
```

第一版先不把它作为硬规则，只用于观测和验收。

后续可以增加规则：

```text
如果吸收新 cell 后 dispersion 超过上限，则拒绝吸收，改为新建 cluster。
```

### 6.8 欧氏半径公式

如果后续需要兼容 BIRCH 原始半径概念，可以计算：

```text
radius^2 = SS / N - ||LS / N||^2
```

但第一版不建议用它做主阈值。

原因：

```text
1. embedding 检索主路径使用 cosine。
2. 向量通常接近单位长度。
3. cosine 相似度对语义 embedding 更直观。
```

## 7. 数据库改造

当前 `cluster` 表已有：

```text
cluster_id
owner_id
kind
label
centroid_json
member_count
last_ts
preview
updated_at
```

第一版建议新增：

```text
algorithm_version TEXT NOT NULL DEFAULT 'micro_cluster_v1'
created_at TEXT
ls_vector_json TEXT NOT NULL DEFAULT '[]'
ss_scalar REAL NOT NULL DEFAULT 0.0
cohesion REAL
dispersion REAL
```

字段含义：

```text
ls_vector_json = BIRCH CF 里的 LS
ss_scalar      = BIRCH CF 里的 SS，保存 sum(||x||^2)
cohesion       = ||LS|| / N
dispersion     = 1 - cohesion
```

`cluster_member` 表当前已有：

```text
cluster_id
member_type
member_id
cell_id
score
```

第一版复用 `score`，但语义改为：

```text
score = cell 分配进 cluster 时的 cosine similarity
```

如果是新建 cluster 的第一个成员：

```text
score = 1.0
```

后续可以再加：

```text
added_at
```

但第一版不是必须。

## 8. CellReplaceResult 改造

当前 `replace_cells_for_entries()` 只返回是否 changed。

需要扩展为：

```python
@dataclass(frozen=True)
class CellReplaceResult:
    changed: bool
    inserted_or_updated: int
    unchanged: int
    deleted: int
    inserted_or_updated_cell_ids: list[str]
    deleted_cell_ids: list[str]
```

或者更直接返回对象：

```python
changed_cells: list[MemoryCell]
deleted_cell_ids: list[str]
```

推荐返回：

```text
changed_cells
deleted_cell_ids
```

因为 cascade 后续做增量聚类时需要完整 cell 内容和 vector。

## 9. 增量聚类流程

入口：

```python
clusterer.update(
    changed_cells=replace_result.changed_cells,
    deleted_cell_ids=replace_result.deleted_cell_ids,
)
```

流程：

```text
1. 删除旧 cell：用 CF 统计量从原 cluster 中扣减。
2. 新增新 cell：找最近 cluster，并用 CF 统计量吸收或新建。
3. 如果 CF 状态异常，再对受影响 cluster 做局部重算修复。
```

### 9.1 删除旧 cell：CF 扣减

```text
deleted_cell_ids
  ↓
查 cluster_member，拿 old_cluster_id
  ↓
读取旧 cell 的 vector_json
  ↓
从 cluster 的 N / LS / SS 中扣减
  ↓
删除 cluster_member
```

对每个被删除的 cell：

```text
N'  = N - 1
LS' = LS - x
SS' = SS - ||x||^2
```

如果：

```text
N' <= 0
```

则删除空 cluster：

```text
DELETE FROM cluster_member WHERE cluster_id = ?
DELETE FROM cluster WHERE cluster_id = ?
```

否则更新 cluster：

```text
centroid' = LS' / N'
cohesion' = ||LS'|| / N'
dispersion' = 1 - cohesion'
```

`last_ts` 和 `preview` 有一个细节：如果删除的不是最新成员，可以保持不变；如果删除的是最新成员或 preview 成员，则需要从该 cluster 的剩余 members 里局部刷新：

```text
refresh_cluster_metadata(cluster_id):
  last_ts = max(remaining.created_at)
  preview = 最近 5 条 text[:200]
```

也就是说，删除路径的主逻辑仍然是 CF 增量扣减；只有 `last_ts / preview` 这种展示型字段需要按 cluster 局部读取。

### 9.2 新增 cell：CF 吸收或新建

对每个 changed cell：

```text
如果 kind 不在 CLUSTERABLE_KINDS:
  跳过

读取同 owner_id + kind + algorithm_version 的 clusters
  ↓
按 time_window 过滤
  ↓
计算 cosine(cell_vector, normalize(cluster.centroid))
  ↓
选 top1
  ↓
top1.score >= threshold:
    absorb_cell(cluster, cell, score)
否则:
    create_cluster(cell)
```

吸收时：

```text
N'  = N + 1
LS' = LS + x
SS' = SS + ||x||^2
centroid' = LS' / N'
cohesion' = ||LS'|| / N'
dispersion' = 1 - cohesion'
```

然后写入成员关系：

```sql
INSERT INTO cluster_member(cluster_id, member_type, member_id, cell_id, score)
VALUES (?, 'memcell', ?, ?, ?)
```

这里的 `score` 是：

```text
score = cosine(cell_vector, normalize(old_centroid))
```

新建 cluster 时：

```text
cluster_id = "cluster_" + cell.cell_id
N = 1
LS = x
SS = ||x||^2
centroid = x
cohesion = ||x|| / 1
dispersion = 1 - cohesion
preview = cell.text[:200]
score = 1.0
```

### 9.3 修改 cell

Markdown entry 修改时，旧 cell 和新 cell 的内容、hash、embedding 都可能变化。

因此修改不要在原 cluster 里原地更新，而是拆成两步：

```text
remove old cell from old cluster
assign new cell to best cluster
```

这样可以处理：

```text
1. 修改后仍属于原 cluster。
2. 修改后应该进入另一个 cluster。
3. 修改后应该新建 cluster。
```

### 9.4 局部重算：异常恢复路径

局部重算不是日常主路径，而是 repair fallback。

只在这些情况触发：

```text
1. cluster 的 N 和 cluster_member 数量对不上。
2. ls_vector_json 缺失或维度不一致。
3. ss_scalar 异常，例如小于 0。
4. cluster_member 指向的 memcell 不存在。
5. 维护命令显式要求 repair cluster。
```

局部重算只处理 `affected_cluster_ids`，不是全量重聚类。

`recompute_cluster(cluster_id)` 的读取逻辑是：

```sql
SELECT
  m.cluster_id,
  m.cell_id,
  c.kind,
  c.text,
  c.created_at,
  c.vector_json
FROM cluster_member m
JOIN memcell c ON c.cell_id = m.cell_id
WHERE m.cluster_id = ?
ORDER BY c.created_at, c.cell_id
```

如果没有剩余有效成员：

```text
DELETE FROM cluster_member WHERE cluster_id = ?
DELETE FROM cluster WHERE cluster_id = ?
```

如果还有成员，则从成员向量重建 CF：

```text
N  = len(valid_members)
LS = sum(member_vectors)
SS = sum(||vector||^2 for vector in member_vectors)
centroid = LS / N
cohesion = ||LS|| / N
dispersion = 1 - cohesion
```

这里的 valid member 指：

```text
cluster_member 存在
memcell 也存在
vector_json 可解析
向量维度正确
```

### 9.5 last_ts 怎么算

`last_ts` 取剩余成员里最大的 `created_at`：

```text
last_ts = max(member.created_at)
```

注意这里保存的是 ISO 时间字符串，而聚类比较时会转成 epoch ms。

### 9.6 preview 怎么生成

preview 不是完整文本，只是 cluster 的可读预览。

第一版规则：

```text
按 created_at 升序排序
取最后 5 个成员
每个成员取 text[:200]
用换行拼接
最终最多保留 500 字符
```

伪代码：

```python
preview_items = [member.text[:200] for member in sorted_members[-5:]]
preview = "\n".join(preview_items)[-500:]
```

这样 cluster 的 preview 更偏向最近内容，符合长期记忆“近事更重要”的直觉。

### 9.7 cluster_member.score 要不要重算

日常增量路径里，新增成员的 `score` 写入分配时的相似度：

```text
score = cosine(cell_vector, normalize(old_centroid))
```

如果触发 `recompute_cluster()`，可以同步更新所有剩余成员的 score：

```text
score = cosine(member_vector, new_centroid)
```

这样 `cluster_member.score` 始终表示：

```text
当前成员和当前 cluster centroid 的相似度
```

而不是“当初被分配进来时的相似度”。

对应更新逻辑：

```text
DELETE FROM cluster_member WHERE cluster_id = ?
重新 INSERT 所有剩余成员，score 写 cosine(member_vector, new_centroid)
```

这样实现简单，也和当前 `upsert_cluster()` 的覆盖式写入风格一致。

### 9.8 是否自动拆分 cluster

第一版不做自动拆分。

也就是说，如果一个 cluster 删除若干成员后内部变得不紧密，第一版不会自动拆成多个 cluster。

原因：

```text
1. 自动拆分需要更复杂的簇内聚类算法。
2. 删除/修改通常只影响少量成员，先保证工程路径稳定。
3. 如果怀疑 cluster 污染，可以通过 recluster_all() 全量重建。
```

后续可以增加维护命令：

```text
recluster_dirty_clusters()
```

专门处理低内聚 cluster。

## 10. 阈值建议

第一版先用固定配置，不做自适应：

```text
episode:      threshold=0.65, time_window_days=7
atomic_fact:  threshold=0.70, time_window_days=30
foresight:    threshold=0.70, time_window_days=30
agent_case:   threshold=0.70, time_window_days=90
agent_skill:  threshold=0.75, time_window_days=365
idea_capture: threshold=0.68, time_window_days=90
```

说明：

```text
episode 时间性强，窗口短一些。
atomic_fact / foresight 可以跨更长时间。
idea_capture 往往围绕长期项目，窗口更长。
agent_skill 更像长期经验，窗口最长。
```

## 11. 冷启动与迁移

增量聚类需要处理几个边界情况。

### 11.1 新系统没有 cluster

如果：

```text
memcell 已经有 vector
cluster 表为空
```

则第一个可聚类 cell 直接新建 cluster：

```text
N = 1
LS = cell.vector
SS = ||cell.vector||^2
centroid = cell.vector
```

这属于正常冷启动，不需要特殊兜底。

### 11.2 旧库没有 LS / SS

如果旧库已经有：

```text
cluster
cluster_member
centroid_json
```

但没有：

```text
ls_vector_json
ss_scalar
algorithm_version
```

不能直接做增量更新。

启动或升级时应该执行一次：

```text
repair_clusters()
```

流程：

```text
读取 cluster_member
  ↓
JOIN memcell.vector_json
  ↓
重新计算 N / LS / SS / centroid
  ↓
回填 cluster 表
```

如果某个 cluster 无法修复：

```text
成员 cell 不存在
vector_json 不存在
向量维度不一致
```

则删除该 cluster，或让 `recluster_all()` 从 memcell 全量重建。

因为 cluster 是派生状态，删除后可重建。

### 11.3 cell 没有 vector

如果某个 cell 没有 `vector_json`：

```text
不能参与聚类
```

cascade 顺序必须保证：

```text
upsert_cell()
  ↓
embedder.embed()
  ↓
写 memcell.vector_json
  ↓
写 LanceDB
  ↓
cluster assign
```

如果 embedding 失败：

```text
本次 cascade job 应失败或重试
不应该创建无向量 cluster
```

### 11.4 embedding model / algorithm version 不一致

cluster 表需要记录：

```text
embedding_model
algorithm_version
```

如果当前配置和 cluster 记录不一致：

```text
不做增量更新
触发 repair 或 recluster_all
```

典型场景：

```text
embedding 模型从 A 换到 B
向量维度变化
threshold 调整
算法从 micro_cluster_v1 升级到 micro_cluster_v2
```

## 12. 数据规模与文件增长策略

长期运行时，数据增长压力主要来自历史材料和向量索引，不是 cluster 本身。

### 12.1 Markdown 文件增长

当前 Markdown 路径规则：

```text
episode:
  users/<user_id>/<session_id>/ep-YYYY-MM-DD.md

atomic_fact:
  users/<user_id>/.atomic_facts/atomic_fact-YYYY-MM-DD.md

foresight:
  users/<user_id>/.foresights/foresight-YYYY-MM-DD.md

agent_case:
  agents/<user_id>/.cases/agent_case-YYYY-MM-DD.md

user_profile:
  users/<user_id>/user.md

agent_skill:
  agents/<agent_id>/skills/skill_<name>/SKILL.md
```

增长最快的是：

```text
episode
atomic_fact
session jsonl
LanceDB vector index
SQLite memcell.vector_json
```

长期也不会很多的是：

```text
user_profile
agent_skill
cluster
conversation_status
unprocessed_buffer
md_change_state
```

### 12.2 哪些内容写 Markdown

Markdown 只存：

```text
人能读
人能改
能作为 source of truth
算法变化后能重建派生状态
```

适合写 Markdown：

```text
episode
atomic_fact
foresight
agent_case
user_profile
agent_skill
idea_capture
```

不适合写 Markdown：

```text
cluster
cluster_member
N / LS / SS
ome_jobs
md_entry_state
sync queue
LanceDB index metadata
```

这些是算法状态或派生索引，应该留在 SQLite / LanceDB。

### 12.3 cluster 不写 Markdown

cluster 是派生状态：

```text
memcell / episode / fact
  ↓
cluster
  ↓
OME 使用 cluster 做高层派生
```

它不是事实源。

因此：

```text
cluster 不写 Markdown
cluster_snapshot 第一版不做
cluster 可从 memcell 全量重建
```

这样可以避免文件数量因为聚类算法增长而爆炸。

### 12.4 长期维护策略

建议后续增加：

```text
repair_clusters()
  不改变成员归属，只重算 N / LS / SS / centroid。

recluster_all()
  清空 cluster，从 memcell 重新决定成员归属。

archive_sessions()
  归档较老 session jsonl。

dedupe_atomic_facts()
  对 atomic_fact 做去重或合并。
```

其中：

```text
repair_clusters = 校准统计量
recluster_all   = 重新聚类
```

这两个不能混淆。

## 13. LLM 聚类策略

第一版默认关闭 LLM 聚类。

原因：

```text
1. token 成本不可控。
2. 当前 agent_case 数据还少。
3. 增量聚类第一版先追求稳定和可验收。
```

后续可选策略：

```text
agent_case:
  similarity >= 0.78 -> 直接并入
  similarity <= 0.50 -> 直接新建
  0.50 < similarity < 0.78 -> 可选 LLM refine
```

LLM refine 应该只在维护任务或显式配置打开时运行。

## 14. 和 OME 的关系

OME 当前会读取 cluster，找相关成员做 profile / skill 抽取。

第一版保持 OME 读取方式不变：

```text
cluster -> cluster_member -> memcell
```

增量聚类只改变 cluster 的维护方式，不改变 OME 的抽取接口。

后续可以升级成：

```text
先找 top clusters
再取 top cluster members
再交给 extractor
```

但第一版不做。

## 15. 全量重建

保留全量重建能力：

```python
rebuild_clusters()
```

行为：

```text
DELETE FROM cluster;
DELETE FROM cluster_member;
读取所有 clusterable memcell;
按 created_at 排序;
逐个 assign_cell();
```

触发场景：

```text
embedding 模型变化
threshold 变化
algorithm_version 变化
手动大规模改 Markdown
怀疑 cluster 被污染
调试 / 验收
```

## 17. 验收标准

第一版完成后，需要能验证：

```text
1. 修改一个 Markdown entry，只重新处理对应 entry 的 cells。
2. 修改一个 cell 后，只影响相关 cluster，不全量重聚类。
3. 删除 entry 后，对应 cluster_member 被删除。
4. 空 cluster 会被删除。
5. cluster_member.score 不再全是 1.0。
6. memcell 默认不再参与聚类。
7. /memory rebuild 或内部 rebuild 仍能全量重建 cluster。
8. 检索主路径不受影响。
9. OME 仍能从 cluster 中读取候选。
```

```text
Mira 第一版不完整复现 CluStream，而是吸收它的 online micro-cluster 思想：日常只做增量更新，必要时从 Markdown / memcell 全量重建。
```
