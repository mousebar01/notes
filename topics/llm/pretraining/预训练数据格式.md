# 大模型预训练数据的几种常见格式

大模型预训练数据本质上是大量文本和元信息的集合。不同数据格式的差别，主要体现在：

- 是否方便人工查看
- 是否支持复杂结构
- 是否适合流式读取
- 是否适合大规模压缩和训练读取

常见格式可以先这样理解：

| 格式 | 特点 | 适合场景 |
| --- | --- | --- |
| CSV | 简单、可读，像表格 | 小规模表格数据、人工查看 |
| JSON | 结构灵活，支持嵌套 | API、配置、结构化样本 |
| JSONL | 一行一个 JSON，方便逐行读取 | NLP 数据集、指令数据、日志式数据 |
| Parquet | 二进制、列式、压缩好 | 大规模数据集、预训练数据 |

如果只是几千条样本，CSV、JSON、JSONL 都够用。但预训练数据通常非常大，可能是几百 GB、几 TB，甚至更多。这时重点就不只是“能不能存”，而是“能不能高效读取、压缩和分块处理”。

## JSONL：训练数据里很常见的文本格式

JSONL 是一行一个 JSON 对象，例如：

```jsonl
{"text": "今天学习大模型", "source": "web"}
{"text": "Transformer 是一种架构", "source": "book"}
{"text": "LoRA 用于参数高效微调", "source": "paper"}
```

它的好处是简单、直观、容易追加，也方便一行一行读取。很多微调数据、指令数据、评测数据都会用 JSONL。

但它的问题也明显：数据量很大时，文本格式体积偏大；如果只想读其中某个字段，也通常还是要解析整行 JSON。

## Parquet：更适合大规模训练数据

Parquet 是一种数据文件格式，全名一般叫 **Apache Parquet**。它适合存储大规模表格数据。

逻辑上，它存的内容也可以像普通表格一样：

| id | text | source |
| --- | --- | --- |
| 1 | 今天学习大模型 | web |
| 2 | Transformer 是一种架构 | book |
| 3 | LoRA 用于参数高效微调 | paper |

区别在于，Parquet 不是普通文本文件，而是二进制格式。它更适合压缩、分块和高效读取。

## Parquet 的核心特点：列式存储

CSV、JSONL 更接近“按行存储”。而 Parquet 是按列存储：

```text
id 列：1, 2, 3, ...
text 列：今天学习大模型, Transformer 是一种架构, ...
source 列：web, book, paper, ...
```

这对训练数据很有用。比如训练时只需要读取 `text` 这一列，就可以尽量避免把 `id`、`source` 或其他元信息全部读出来。

常见读取方式类似这样：

```python
import pyarrow.parquet as pq

pf = pq.ParquetFile(filepath)
row_group = pf.read_row_group(row_group_idx)
texts = row_group.column("text").to_pylist()
```

这段代码大致意思是：

```text
打开一个 parquet 文件
↓
读取其中一个 row group
↓
取出 text 这一列
↓
转成 Python list
```

## Row group：按块读取

Parquet 文件内部还会分成一块一块的数据，叫 **row group**。

可以理解成：

```text
一个 parquet 文件
├── row group 0
├── row group 1
├── row group 2
└── ...
```

每个 row group 里面有一批行。读取时不需要一次把整个大文件读进内存，而是可以一次读一个 row group。

这对大模型训练很重要，因为训练数据太大，不可能一次性全部加载到内存里。

## 我的理解

如果从大模型训练数据的角度看，可以粗略这样分：

- JSONL 更适合中小规模数据、人工检查、指令微调数据。
- Parquet 更适合大规模预训练数据，因为它压缩好、读取快，支持按列和按块读取。
- CSV 简单但表达能力弱，更适合作为小表格或临时数据。
- JSON 结构灵活，但大规模逐条读取时不如 JSONL 顺手。

一句话总结：

> Parquet 是一种适合存储和读取大规模表格数据的高效文件格式。大模型预训练数据常用它来保存海量文本，因为它压缩好、读取快，支持按列读取和分块读取。
