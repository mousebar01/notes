# 大模型推理与生成系统设计

此笔记梳理了大模型推理（Inference）与文本生成中的核心设计要点。相比于训练阶段，推理阶段更加关注生成的速度、显存消耗、可控性与可用性。

## 1. 训练与推理的核心关注点对比

- **训练阶段**：主要关心“如何更快、更稳地降低损失值（Loss）”。
- **推理阶段**：主要关心“如何更快、更省显存、更可控、更好用地生成 Token”。

## 2. 推理系统的核心设计要点


```text
1. 解码策略
```

比如：

```text
temperature=0      # 贪心，选最高概率 token
temperature>0      # 随机采样
top_k=50           # 只从前 k 个候选里采样
top_p              # nucleus sampling，nanochat 这里没写
repetition penalty # 降低重复生成，nanochat 这里也没写
```

这些会直接影响生成风格：稳定、发散、重复、创造性。

```text
2. KV cache
```

这是推理加速的核心。没有 KV cache，每生成一个 token 都要重新算整个上下文；有 KV cache，每步只算新 token。

它影响：

```text
推理速度
显存占用
最大上下文长度
batch 生成效率
```

```text
3. batching
```

nanochat 的 `generate(tokens, num_samples=...)` 可以一次生成多个 samples。实际服务里还会做 continuous batching，把不同用户请求拼在一起跑，提高 GPU 利用率。

```text
4. stopping condition
```

什么时候停止生成？

```text
达到 max_tokens
生成 <|assistant_end|>
生成 <|bos|>
遇到 stop words
工具调用结束
```

nanochat 里能看到：

```python
if next_token == assistant_end or next_token == bos:
    state.completed = True
```

```text
5. 特殊 token 和对话格式
```

base model 只是续写，chat model 需要格式：

```text
<|user_start|> ... <|user_end|>
<|assistant_start|> ... <|assistant_end|>
```

这会影响模型是否知道“该轮到助手说话”。

```text
6. 工具调用 / tool use
```

nanochat 的 `engine.py` 里已经有简单的 Python/calculator 工具逻辑：

```text
<|python_start|> ... <|python_end|>
<|output_start|> ... <|output_end|>
```

这已经是 agent/tool-use 的雏形。

```text
7. 精度与部署性能
```

推理时可以考虑：

```text
bf16 / fp16
int8 / int4 量化
FlashAttention
PagedAttention
tensor parallel
speculative decoding
```

这些属于推理优化工程。

```text
8. 流式输出
```

训练是一次算 loss，推理通常要边生成边显示：

```text
for token in stream:
    print(token)
```

nanochat 的 `generate()` 就是一个 generator，可以边产 token 边输出。

所以你这个感觉是对的：**推理不是简单地 model(prompt) 一下，它是一整套生成系统设计。**

不过学习顺序上，先抓住这条主线就够：

```text
prompt -> token ids -> logits -> sample next token -> append -> repeat -> decode
```

其它设计点都是围绕这条主线做速度、质量和产品体验优化。
